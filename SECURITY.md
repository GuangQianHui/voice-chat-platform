## 报告漏洞

我们非常重视安全问题。如果您发现了安全漏洞，请通过以下方式报告：

### 私密报告（推荐）

- **邮箱**: security@yourdomain.com
- **PGP 密钥**: [下载公钥](https://yourdomain.com/security.asc)
- **加密邮件**: 使用我们的 PGP 公钥加密邮件

### 公开报告

- **GitHub Issues**: 创建带有 `[SECURITY]` 标签的 issue
- **安全公告**: 在 [Security Advisories](https://github.com/yourusername/voice-assistant/security/advisories) 中报告

## 报告格式

请包含以下信息：

```
**漏洞类型**: [例如 XSS, CSRF, 信息泄露等]
**影响版本**: [受影响的版本号]
**漏洞描述**: [详细描述漏洞]
**重现步骤**: [如何重现漏洞]
**预期行为**: [正常情况下的行为]
**实际行为**: [漏洞触发时的行为]
**影响评估**: [漏洞的严重程度和影响范围]
**修复建议**: [如果有的话]
```

## 响应时间

我们承诺在以下时间内响应安全报告：

- **初步确认**: 24 小时内
- **详细评估**: 72 小时内
- **修复计划**: 7 天内
- **修复发布**: 根据严重程度，1-30 天

## 严重程度分级

我们使用以下分级标准：

### 🔴 严重 (Critical)

- 远程代码执行
- 身份验证绕过
- 数据泄露（敏感信息）
- 影响所有用户的核心功能

### 🟠 高危 (High)

- 权限提升
- 数据篡改
- 拒绝服务攻击
- 影响部分用户的重要功能

### 🟡 中危 (Medium)

- 信息泄露（非敏感）
- 跨站脚本攻击
- 跨站请求伪造
- 影响用户体验的功能

### 🟢 低危 (Low)

- 信息泄露（公开信息）
- 拒绝服务（轻微）
- 界面问题
- 不影响功能的问题

## 安全最佳实践

### 对于用户

1. **保持更新**: 及时更新到最新版本
2. **安全配置**: 使用 HTTPS 和安全的配置
3. **权限管理**: 最小化应用权限
4. **监控日志**: 定期检查应用日志

### 对于开发者

1. **代码审查**: 所有代码变更需要安全审查
2. **依赖管理**: 定期更新依赖包
3. **安全测试**: 集成安全测试到 CI/CD 流程
4. **文档更新**: 及时更新安全相关文档

## 安全更新

- 安全更新会通过 GitHub Releases 发布
